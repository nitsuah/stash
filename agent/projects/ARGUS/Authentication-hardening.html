Quick Guide

  1. Clone the Odysseus repo.
  2. Copy and edit the .env file (admin password, auth, GPU type, model choices).
  3. Choose the proper Docker‑Compose file (standard or GPU overlay) and bring the stack up.
  4. Harden the deployment (HTTPS reverse‑proxy, secure cookies, disable raw model ports).
  5. Mount a host‑directory as ./skills/ and bulk‑copy or import skill packs.
  6. Restart (or run reload_skills) so Odysseus indexes the new skills.
  7. (Optional) Attach an external vector store (Qdrant/Milvus) and point VECTOR_DB_URL.
  8. Verify everything works, then set up a CI pipeline to keep skills and the core stack up‑to‑date.

Detailed Steps

## Step 1 – Clone the repository and prepare the environment

git clone [https://github.com/pewdiepie-archdaemon/odysseus.git](https://github.com/pewdiepie-archdaemon/odysseus.git)
cd odysseus
cp .env.example .env

Why: Cloning the main branch guarantees you have the latest stable code and the .env.example file lists all configurable variables, preventing “missing key” errors at start‑up.

Tip: If you plan to contribute back, fork the repo first and clone your fork.

## Step 2 – Edit the .env file for security and hardware

Open .env in your favourite editor and adjust the following sections (values are examples; replace with your own secrets):

ADMIN_PASSWORD=SuperSecret123!          # change immediately
AUTH_ENABLED=true                       # enable multi‑user login
SECURE_COOKIES=true                     # forces Secure + HttpOnly flags
EXPOSE_MODEL_PORTS=false                # hide raw model APIs from the internet

# Model defaults (choose according to hardware)
DEFAULT_CHAT_MODEL=gpt-4o-mini         # cloud model for normal chats
UTILITY_MODEL=mistral-tiny              # lightweight model for embeddings, summaries
VISION_MODEL=clip-vit-b32               # optional, enable only if you need image analysis

# GPU selection (choose one)
GPU_TYPE=nvidia                         # or 'amd' or 'cpu'
# If you have multiple GPUs, you can later set NVIDIA_VISIBLE_DEVICES=0,1

# Persistence
DATABASE_URL=sqlite:///data/odysseus.db # for small setups; switch to PostgreSQL for production
VECTOR_DB_URL=[http://qdrant:6333](http://qdrant:6333)        # point to external vector store if you have one
PERSISTENT_MEMORY=true                  # keep vector memory across restarts

# Miscellaneous
UPLOAD_LIMIT_MB=100                     # caps file‑upload size (helps when bulk‑importing)
COOKBOOK_AUTO_DOWNLOAD=true             # auto‑fetch compatible models

Why:
- Auth & cookies stop unauthorised users from hijacking the UI.
- EXPOSE_MODEL_PORTS=false prevents attackers from calling the raw Ollama/vLLM endpoints directly.
- GPU_TYPE tells the compose overlay which runtime flags to inject.
- VECTOR_DB_URL moves the vector store out of the container, allowing you to store millions of embeddings without exhausting local disk.

Warning: Never commit the .env file to a public repo; it contains secrets.

## Step 3 – Choose the appropriate Docker‑Compose file

  • CPU‑only / default:

bash docker compose up -d

  • GPU‑accelerated (NVIDIA):

bash COMPOSE_FILE=docker-compose.yml:docker-compose.gpu.yml \ GPU_TYPE=nvidia \ docker compose up -d

The docker-compose.gpu.yml overlay adds:

yaml runtime: nvidia environment: - NVIDIA_VISIBLE_DEVICES=all - NVIDIA_DRIVER_CAPABILITIES=compute,utility

  • AMD GPU: replace the overlay with docker-compose.amd.yml (see the community discussion for exact flags).

Why: The overlay injects the necessary driver capabilities; without it the containers will fall back to CPU, dramatically increasing latency for 70B‑class models.

Tip: Run docker compose config to see the final merged configuration before launching.

## Step 4 – Harden the network with a reverse‑proxy

Deploy a lightweight reverse‑proxy (Caddy is a popular choice because it auto‑generates Let’s Encrypt certificates).

Caddy example (Caddyfile):

odysseus.example.com {
    reverse_proxy odysseus:8080
    encode gzip
    tls you@example.com
}

Run it alongside Odysseus:

docker run -d \
  -p 80:80 -p 443:443 \
  -v $(pwd)/Caddyfile:/etc/caddy/Caddyfile \
  -v caddy_data:/data \
  --name odysseus-proxy \
  caddy:latest

Why:
- HTTPS is required for SECURE_COOKIES=true to take effect.
- The proxy isolates the UI (port 8080) from the host network, reducing the attack surface.

## Step 5 – Prepare the skills directory

Create a host folder that will be mounted into the container:

mkdir -p /opt/odysseus/skills

5.1 Bulk copy from a local collection

If you already have a folder with thousands of skill files:

cp -r /path/to/my‑skill‑bundle/* /opt/odysseus/skills/
docker compose restart odysseus   # forces re‑indexing

Odysseus scans the skills/ directory every 30 seconds; a restart makes the change immediate.

5‑2 Import from public GitHub skill packs (recommended for massive imports)

Install the community importer:

pip install autoskills   # or pull the Docker image: ghcr.io/rodrigo-devv/autoskills

Add a pack:

autoskills add [https://github.com/awesome‑skills/finance‑pack](https://github.com/awesome‑skills/finance‑pack)

The tool:

  • Performs a shallow clone (depth 1) to keep network usage low.
  • Validates each manifest.json for required fields (inputs, outputs, tags).
  • Places the files under /opt/odysseus/skills/imported/finance‑pack.

Tip: Run autoskills verify after import to run the built‑in test harness on every new skill.

5‑3 Version‑controlled skill repo (CI‑friendly)

Create a separate Git repo that only contains skill files. Then mount it directly:

services:
  odysseus:
    volumes:
      - /opt/odysseus/skills:/app/odysseus/skills
      - ./my‑skills-repo:/app/odysseus/skills   # overlay; Git repo lives here

Set up a CI pipeline (GitHub Actions, GitLab CI) that:

  1. Lints each skill (flake8 / pylint).
  2. Runs tests/run_skill.sh <skill> to ensure the expected JSON schema.
  3. Pushes the updated repo; the running container picks up changes on the next 30‑second scan or after a manual docker compose restart odysseus.

Why: Decoupling skills from the core code lets you push updates without rebuilding the Odysseus image, which is crucial when you have thousands of skills that change frequently.

## Step 6 – (Optional) Attach an external vector store for scalable memory

If you anticipate more than a few hundred embeddings, spin up Qdrant:

services:
  qdrant:
    image: qdrant/qdrant:latest
    ports:
      - "6333:6333"
    volumes:
      - qdrant_data:/qdrant/storage
volumes:
  qdrant_data:

Then, in .env:

VECTOR_DB_URL=[http://qdrant:6333](http://qdrant:6333)
PERSISTENT_MEMORY=true

Odysseus will automatically use this endpoint for all embedding‑based memory calls.

Why: The default SQLite store lives in the container’s filesystem and caps out at a few hundred MB; Qdrant can handle tens of millions of vectors with sub‑millisecond ANN search.

## Step 7 – Verify the installation

  1. Open your browser at [https://odysseus.example.com.](https://odysseus.example.com`.)
  2. Log in with the admin credentials you set.
  3. Navigate to Library → Skills – you should see the imported skill list, grouped by any namespace folders you created.
  4. Start a new chat, select a skill from the autocomplete, and watch the response.
  5. (Optional) Open the reverse‑proxy logs (docker logs odysseus-proxy) and Odysseus logs (docker logs odysseus) to confirm no errors.

Tip: Enable LOG_LEVEL=debug temporarily if a skill fails to load; the debug output will show the exact import error.

## Step 8 – Automate updates (CI/CD)

Create a simple GitHub Action that runs nightly:

name: Update Skills & Core

on:
  schedule:
    - cron: '0 2 * * *'   # 02:00 UTC daily
  workflow_dispatch:

jobs:
  update:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout Odysseus
        uses: actions/checkout@v3
        with:
          repository: pewdiepie-archdaemon/odysseus
          ref: main

      - name: Pull latest Docker images
        run: |
          docker compose pull
          docker compose up -d --no-deps --build odysseus

      - name: Pull skill repo
        run: |
          git clone [https://github.com/awesome-skills/finance-pack.git](https://github.com/awesome-skills/finance-pack.git) /opt/odysseus/skills/finance
          docker compose exec odysseus python manage.py reload_skills

Why: Keeping both the core stack and the skill collection up‑to‑date reduces the chance of incompatibilities (e.g., a skill expecting a newer model API).

Common Mistakes

Mistake Consequence How to avoid / fix
Leaving EXPOSE_MODEL_PORTS=true Raw model endpoints become reachable from the internet, exposing your API keys and allowing unauthorised inference. Set the flag to false and verify with docker compose port odysseus 11434 (should show “no such port”).
Running the container as root Increases impact of a potential code injection via a malicious skill. Ensure the Dockerfile sets USER odysseus and keep read_only: true where possible.
Storing .env in the repo Secrets get leaked if the repo is ever made public. Add .env to .gitignore; use Docker secrets or a vault for production.
Importing a skill pack without manifest validation Broken JSON schemas cause runtime errors that crash the UI. Always run autoskills verify or the built‑in test harness before mounting.
Using a GPU‑enabled compose file on a CPU‑only host Containers fail to start with “NVIDIA driver not found”. Verify GPU presence (nvidia-smi) before selecting the GPU overlay; keep a CPU‑only compose file as fallback.
Relying on the default SQLite DB for production Data loss on container recreation; limited concurrency. Switch to PostgreSQL (DATABASE_URL=postgresql://user:pass@db/odysseus) or an external managed DB.
Forgetting to restart after bulk skill copy New skills are invisible until the next automatic scan (up to 30 s) or a manual restart, leading to confusion. Run docker compose restart odysseus or docker exec odysseus python manage.py reload_skills.
Setting UPLOAD_LIMIT_MB too low Large skill bundles (> 50 MB) are rejected, causing incomplete imports. Raise the limit to at least 100 MB for bulk imports, then lower it again if needed.
Not enabling HTTPS SECURE_COOKIES=true has no effect; session cookies travel in clear text. Deploy a reverse‑proxy with Let’s Encrypt before exposing the UI to the internet.
Skipping vector‑store configuration when using many embeddings Memory exhaustion inside the container, leading to OOM kills. Deploy Qdrant/Milvus and point VECTOR_DB_URL to it.
